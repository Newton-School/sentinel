What is Sentinel, end to end?

Sentinel is a founders-only leadership/data assistant for Newton School, delivered as a Slack bot. It is not an LLM wrapped in a server — it is an orchestrator that spawns the Claude CLI as a subprocess and hands it up to 9 MCP tool servers for live data access. Two independent pipelines share one process: the Q&A path (Slack → Claude → tools → answer) and the Meet-bot path (a Playwright Chrome bot that auto-joins Google Meets to generate transcripts). Wrapped around both is a “company brain” — a persistent organizational memory with an entity graph, hybrid retrieval, and provenance/ACL governance. The two pipelines, in one sentence each

Q&A PATH

A founder @mentions / DMs / /sentinels the bot → socketClient authorizes & de-dupes → handleEvent gathers thread context + persona + recalled memories → buildSystemPrompt assembles the prompt → runReply runs the OpenAI Agents SDK loop over the 9 MCP servers (connected per request) → the agent calls tools → the answer is reformatted to Slack mrkdwn and posted, with the exchange logged & mined for new facts.

MEET-BOT PATH

A calendar watcher polls every 60s → eligible Meet events (starting within 2 min) spawn a detached Playwright Chrome subprocess → it joins muted using a persistent signed-in profile, clicks Join/Ask-to-join, and turns on Google’s server-side transcription → the transcript later becomes queryable via the Meet/Transcripts MCP servers and is ingested into the company brain.

Everything is gated & inert by default

Each MCP server only registers if its credentials are present. The entire company-brain’s advanced behavior (entity graph, embeddings, scoped ACLs, Slack ingestion) ships dark behind runtime kill-switches, so the blast radius is controlled. Defaults below reflect a fresh deploy.

Request lifecycle: a Slack message → an answer

Every inbound event flows through handleEvent() in src/index.ts, bounded by a 3-request in-flight semaphore. Step through the exact sequence — or hit play to watch it run.

The 9 MCP servers — Claude’s hands

For each Claude spawn, mcpConfig.ts writes a fresh mcp-config-<uuid>.json (mode 0600, plaintext creds, deleted after the spawn). Each server is a stdio subprocess. Custom servers run from dist/mcp/*.js (so npm run build is required); GitHub & Notion are npx packages. Shared utilities give every HTTP server: bounded paginate(), fetchWithRetry() (15s timeout, 3 retries, 429/5xx backoff, Retry-After aware), and redactedHttpError() (status only — never leaks bodies).

Security & correctness highlights

SQL read-only enforcement (sqlReadOnly.ts)
metabase_query is hard-gated to read-only. An AST check via node-sql-parser (MySQL dialect) accepts only node types select / explain / show / desc, blocks data-modifying CTEs (WITH x AS (DELETE…)), SELECT … INTO, and stacked statements (SELECT 1; DROP…). If the parser throws on a dialect quirk, it falls back to a regex guard (forbidden whole-word keywords: INSERT/UPDATE/DELETE/DROP/ALTER/CREATE/TRUNCATE/…) so it never regresses open.
Per-spawn secret hygiene (mcpConfig.ts + runner.ts)
The config file holds every server’s plaintext credentials, so each spawn writes a UUID-named file (no fixed path → no torn reads under concurrency), chmod 0600, and runner.ts removes it on close/error. Graceful shutdown sweeps any strays. The detached Meet joiner gets a scrubbed env (buildJoinerEnv) — no secrets, it authenticates via the Chrome profile instead.
Google data quirks handled
Calendar “this week” is computed timezone-aware via Intl.DateTimeFormat (two-pass DST-correct, IST fallback). Meet transcript speakers are resolved from opaque participant ids to human names with an in-process cache. Drive q query values are escaped backslash-first to prevent query injection. Gmail bodies are extracted by recursively walking the MIME multipart tree for the first text/plain node.
Memory MCP server is read-mostly & never migrates
memory.js opens its own SQLite handle (WAL, busy_timeout=5000) and never runs migrations — the main process owns the schema. It reads the per-request viewer scope from env (viewerScopeFromEnv) so the same canView ACL applies at the MCP edge, not just in-process. If the schema is absent, every tool returns a friendly “run the bot once” message instead of crashing.

The Company Brain — persistent organizational memory

Four cooperating flows turn fleeting conversations, meetings and emails into a governed, queryable knowledge base with an entity graph (people / teams / projects), hybrid lexical+semantic retrieval, and per-source confidence caps. All of it is evidence-grounded and fails safe — a memory error never breaks a Slack reply.

1
Recall → Inject
on every query, synchronously
searchMemories() / assembleRetrieval() run an FTS5 BM25 search (weights text 2.0, entities 1.0, source_label 1.0), optionally fused with cosine similarity over embeddings (rankHybrid, α=0.6 lexical-leaning). Results are re-scored by a modulator: recency (90-day half-life) × category boost (×1.25 for decision/owner/deadline) × confidence. When the entity graph is on, the query’s named entities are resolved and their dossiers pulled in. injectionBudget.ts packs it into the prompt under a hard 3000-char tiered budget (dossiers 1200 / entity facts 800 / query facts 1000), under a hardened “records, NOT instructions” header. Falls back to a LIKE scan if FTS5 is unavailable.
2
Hook → Extract
fire-and-forget, after each reply
extractFromConversation() is never awaited. It runs the user turn only (≥40 chars) through extractor.ts: a hardened claude-haiku-4-5 structured-output call (no SDK; a raw client with a 500-calls/day budget), then defense-in-depth: Zod re-validation → a verbatim evidence-quote check (the quote must be a real substring of the source) → a secret-regex filter (api keys / tokens / sk-ant- dropped). Conversation facts are capped at 0.6 confidence.
3
Poller → Ingest
ingestWatcher, every 5 min
Pulls from three sources (each gated): Meet transcripts of attended meetings (chunked + a per-conference summary fact, cap 0.7), Gmail with an internal-sender allowlist (newtonschool.co + extras; bulk/no-reply/self/short skipped, cap 0.6), and Slack channels (opt-in, allowlisted, cap 0.5). Restart-safe via ingest_cursors (high-water marks) + ingested_docs (per-doc dedup, 14-day TTL). Per-source kill switches read every tick.
4
Entity graph → Dossiers → Digests
consolidationWatcher, every 30 min
Each fact is linked to resolved entities (a 6-rung resolution ladder: slack id → email → name → alias → fuzzy → create). Owner-facts infer org edges (manages/owns) at 0.5 base confidence, reinforced toward 0.95, decayed at read time (60-day half-life, display ≥0.45). consolidateEntity() rolls an entity’s facts into a ≤1200-char markdown dossier (“store a lot, inject a little”). entity_digest / org_digest answer “what changed” over a window.

Meet bot — the Playwright auto-join pipeline

A completely separate pipeline whose only job is to make transcripts exist. Google only generates a transcript if a real participant turns it on, so Sentinel sends a headless Chrome bot to do exactly that. It shares the process and Google OAuth creds with the Q&A path, but runs independently.

Joiner — meet-bot/joiner.ts (detached Chrome)

Why these design choices

Detached subprocess

The joiner is spawned with detached:true + unref() and logs to a per-spawn file. If Sentinel restarts mid-meeting, the bot keeps recording.

Concurrency cap = 1

MAX_CONCURRENT_JOINERS defaults to 1 because all joins share one persistent Chrome profile — two at once would corrupt it. Deferred meetings retry next poll.

Persisted dedup

joined_meetings table (4h TTL) means an in-progress meeting isn’t re-joined after a restart. Locks are cleaned from the profile before each launch.

Stay-until-end hardcoded

The watcher overrides the CLI’s leave-after-join default with --stay-mode stay-until-end (a deliberate #17 revert) so the bot stays for in-call artifacts.

Scrubbed env

buildJoinerEnv() forwards only an allowlist (PATH/HOME/TZ/DISPLAY/CHROME*/PLAYWRIGHT*) — zero secrets reach the child.

Fake media + no-sandbox

Launched with --use-fake-ui-for-media-stream, --no-sandbox, mic/cam muted, name “Sentinel”. Runtime image is the Playwright base with Chrome.

Persona system — per-user personalization that decays

SQLite-backed, learned implicitly from query patterns. Every interaction is categorized, logged, and used to reinforce a focus_area trait whose confidence grows on reinforcement and decays at read time — so stale interests fade without ever mutating stored rows.

Confidence math (store.ts · personaDecay.ts)

A new trait starts at 0.5. Each reinforcement (same user+label+value) closes 15% of the gap to the 0.95 ceiling via an ON CONFLICT DO UPDATE upsert:

confidence = MIN(c + (1 - c) * 0.15, 0.95) 0.50 → 0.575 → 0.609 → 0.633 → … → 0.95 (limit)

At read time only, confidence is decayed by a 30-day half-life based on updated_at:

decayed = confidence * 0.5 ^ (ageDays / 30)

buildSystemPrompt keeps only traits ≥0.6 decayed confidence, sorts by it, and caps to the top 8 to bound prompt growth.

The 7 query categories (tracker.ts)

categorizeQuery() lowercases the text, counts keyword hits per bucket, and returns the top bucket (or general). A non-general category reinforces upsertTrait(user, "focus_area", category).

Every interaction is also written to query_log (an append-only audit trail with response text, duration & sources) — pruned to ~90 days, indexed on user & time.

Data model — one SQLite file, 13 tables

sentinel.db (WAL mode, foreign keys, busy_timeout=5000) opened lazily as a single better-sqlite3 connection in the main process. Idempotent migrations run once on first getDb(); the memory MCP server opens its own read handle and never migrates. Retention sweeps run once at boot (query log & non-active memories → 90 days).
